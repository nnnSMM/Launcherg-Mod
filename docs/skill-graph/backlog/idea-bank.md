# Idea Bank

### Memo editor の軽量化

- Type: `performance`
- 狙い: Memo 画面でだけ読む EasyMDE 本体チャンクをさらに小さくし、メモ初回表示の待ち時間を減らす。
- 根拠: 2026-06-02 の最適化で EasyMDE は初期ロードから外せたが、機能利用時の遅延チャンクとして約 1MB の minified JS が残っている。
- 最初の一歩: 現行 toolbar / autosave / image paste / preview 要件を棚卸しし、より小さい markdown editor か CodeMirror 個別 import への移行可否を比較する。
- 注意: メモの autosave と画像貼り付けは実利用上重要なので、単純な textarea 退化ではなく機能要件を満たす候補に絞る。

### プレイ時間ヒートマップ

- Type: `visibility`
- 狙い: 日ごと・ゲームごとのプレイ傾向を、GitHub風の年次カレンダーで素早く振り返れるようにする。
- 根拠: GitHub/GitLabのコントリビューションカレンダーは日別タイル、クリック詳細、年単位表示で長期活動を俯瞰できる。WakaTime/Toggl/RescueTime系の時間可視化は、合計時間、内訳、時間帯、フィルタが有効。
- 最初の一歩: 全ゲーム合算の年次ヒートマップ、日付クリック時の日別ゲーム内訳、ゲーム別フィルタを実装する。
- 注意: 既存の `collection_element_daily_play_times` を読み取り元にし、手動補正やセッション履歴は初回実装には含めない。

すぐに実装しない改善候補を置く場所です。実装候補に上げる時は [[template-improvement-card]] の形に整理します。

## 候補

### VNDB照合結果の確認・修正UI

- Type: `reliability`
- 狙い: 自動取得したVNDBスクリーンショットや将来のメタデータが、別作品に紐づく事故を減らす。
- 根拠: 現在はゲーム名、ルビ、ブランド、発売日で候補を自動選択しているため、同名作品や移植版で誤照合が起きうる。
- 最初の一歩: ゲーム詳細に「VNDB候補を確認」導線を追加し、上位候補のタイトル、ブランド、発売日、サムネイルを並べて選べるようにする。
- 注意: 既存の `collectionElementId` とスクリーンショットキャッシュの対応を崩さず、手動選択した `vndbId` を優先する。

### VNDBメタデータカード

- Type: `data`
- 狙い: 詳細画面で発売日、ブランド、評価、想定プレイ時間、対応言語、タグなどを補足表示し、ゲーム選びと積みゲー整理を助ける。
- 根拠: VNDB APIはVNの `rating`、`length_minutes`、`tags`、`developers`、`languages` などを取得できる。
- 最初の一歩: 既存のVNDBスクリーンショットキャッシュとは別に、軽量なメタデータキャッシュを作り、詳細画面に読み取り専用カードとして出す。
- 注意: タグはネタバレ度と成人向けカテゴリの扱いがあるため、初期表示は安全側に絞る。

### 公式・ストアリンク補完

- Type: `workflow`
- 狙い: 登録済みゲームから公式サイト、Steam、DLsiteなどの外部ページへ移動しやすくする。
- 根拠: VNDBのリリース情報には `extlinks` があり、リリース単位の外部リンクを取得できる。
- 最初の一歩: VNDB照合済みゲームの日本語リリースを取得し、詳細画面の外部リンク欄に「公式」「Steam」などを小さく表示する。
- 注意: 外部リンクの種類はVNDB側のスキーマ変更を受けるため、未知のリンク名でも壊れない表示にする。

### シリーズ・関連作レーン

- Type: `visibility`
- 狙い: ファンディスク、続編、移植版、前作後作を一覧から見つけやすくする。
- 根拠: VNDBのVNフィールドには `relations` があり、関連作品と関係種別を取得できる。
- 最初の一歩: 詳細画面に「関連作」レーンを追加し、ローカル登録済みなら起動導線、未登録ならVNDBリンクを出す。
- 注意: 作品間の関係はユーザーの所持状況と別なので、自動登録はせず表示だけに留める。

### VNDBユーザーリスト連携

- Type: `research`
- 狙い: VNDB側の投票、プレイ開始日、終了日、ラベルとLauncherg-Modのプレイ状況を、任意で同期できるようにする。
- 根拠: VNDB APIはトークン認証によりユーザーリストの読み取り、更新ができる。
- 最初の一歩: 書き込みなしの読み取り専用インポートから始め、VNDBの `finished` や `labels` をローカルのプレイ状況候補として提示する。
- 注意: APIトークンとプライベートリストを扱うため、明示的なオプトイン、保存場所、削除導線を先に設計する。

### 今夜の1本レコメンド

- Type: `workflow`
- 狙い: ユーザーが「次に何を遊ぶか」で迷う時間を減らし、登録済みゲームの起動につなげる。
- 根拠: ローカルのプレイ履歴、未クリア状態、総プレイ時間にVNDBの `rating`、`length_minutes`、タグを組み合わせると、短時間向け、長編向け、雰囲気別の候補を出せる。
- 最初の一歩: ホーム上部に「今夜の1本」を1件だけ出し、理由を「短め」「評価高め」「最近触っていない」など2つまで表示する。
- 注意: 推薦ロジックは決定的なスコア計算にし、AI要約は理由文の整形に限定する。

### 発見ストア

- Type: `workflow`
- 狙い: 購入ではなく、新しいVNを探して「気になる」「所持済み」「対象外」に分けられる発見画面を作る。
- 根拠: VNDB APIは `rating`、`votecount`、`released`、`length`、`lang`、`platform`、`tag`、`developer`、`release.extlink` などで検索・並び替えできるため、Steamのストア棚に近い体験を構成できる。
- 最初の一歩: 新しい「発見」タブを作り、日本語対応、Windows対応、評価順、新作順、短編、人気タグなどの棚を読み取り専用で表示する。
- 注意: 購入・価格・在庫を扱わず、外部ストアリンクがある場合も「外部で確認」に留める。API結果はキャッシュし、VNDBへの過剰アクセスを避ける。

### ネタバレ安全タグ・メモモード

- Type: `visibility`
- 狙い: 未プレイ作品の詳細画面で、ネタバレになりやすいタグや情報を不用意に見せない。
- 根拠: VNDBのタグには `spoiler` レベルがあり、0から2の段階で扱える。
- 最初の一歩: 詳細画面のVNDBタグ表示を「安全」「軽いネタバレまで」「全表示」の3段階にし、初期値は安全にする。
- 注意: 既存メモやユーザー画像はVNDB由来ではないため、同じ制御対象に混ぜない。

### 似た積みゲー発掘

- Type: `data`
- 狙い: クリア済み・お気に入り作品のタグ構成から、手元にある未プレイ作品を再発見させる。
- 根拠: VNDBタグの重なり、評価、想定プレイ時間を使うと、ローカルライブラリ内だけで類似作品を並べられる。
- 最初の一歩: 作品詳細に「手元の似た作品」を3件だけ表示し、共通タグとプレイ状況を添える。
- 注意: VNDBに登録されていない作品や照合失敗作品が不利になりすぎないよう、通常検索とは分ける。

### ルート分岐しおり

- Type: `workflow`
- 狙い: 複数ルート作品で、どこまで進めたかを作品単位より細かく管理できるようにする。
- 根拠: VNDBのキャラクター、スタッフ、タグ情報を補助に使うと、ルート名やヒロイン名の候補を提示できる。
- 最初の一歩: 詳細画面に任意の「ルートしおり」を追加できるUIを作り、VNDB照合済みならキャラクター名候補を出す。
- 注意: VNDBのキャラクター情報取得は量が増えやすいため、初回は手入力主体にして候補表示だけにする。

### 外部リンク生存チェックと購入先整理

- Type: `reliability`
- 狙い: 古い作品の公式サイト消失や販売終了を見つけ、ユーザーが参照先を整理できるようにする。
- 根拠: VNDBリリースの `extlinks` とローカルキャッシュを比較すれば、公式、Steam、DLsiteなどの現行リンク候補を持てる。
- 最初の一歩: 詳細画面で外部リンク一覧を表示し、リンク切れ確認はユーザー操作時だけ行う。
- 注意: 自動巡回は負荷や規約面の不安があるため、バックグラウンドで大量チェックしない。

### 文字化けしている日本語ドキュメントの復旧計画

- Type: `maintenance`
- 狙い: README、USAGE、コメントの可読性を戻し、ユーザー向け説明と将来の開発効率を上げる。
- 注意: 元のエンコーディングと意図を確認してから範囲を決める。機能変更と混ぜない。

### Windows統合の手動QAチェックリスト

- Type: `reliability`
- 狙い: トレイ、ショートカット、スクリーンショット、プロセス監視の回帰確認を定型化する。
- 最初の一歩: [[quality-gates]] の手動確認項目を `docs` またはSkill Graphの専用ノートへ具体化する。

### インポート処理の失敗理由を見える化する

- Type: `visibility`
- 狙い: フォルダスキャンや手動追加で、何が追加されなかったのかをユーザーが理解できるようにする。
- 最初の一歩: `create_elements_in_pc` からフロントへ返す情報と既存UIの進捗表示を調査する。

### スクリーンショット管理の操作回帰テストを増やす

- Type: `reliability`
- 狙い: 別ウィンドウの初期引数、並び替え、削除、インポートの破損を減らす。
- 注意: Tauri実機確認が必要な範囲と、純粋なロジックテストで守れる範囲を分ける。

### ゲーム詳細画面の情報密度を見直す

- Type: `workflow`
- 狙い: 起動、メモ、スクリーンショット、パス修正、削除などの操作を迷わず行えるようにする。
- 最初の一歩: `src/components/Work` の構造と現行スクリーンショットを確認し、変更なしのUXレビューを行う。

### DB migrationの互換性チェックを強化する

- Type: `data`
- 狙い: 既存DBに対するmigration追加時の事故を減らす。
- 最初の一歩: 現在の `src-tauri/src/migrations` と `launcherg_sqlite.db3` の扱いを調査する。
